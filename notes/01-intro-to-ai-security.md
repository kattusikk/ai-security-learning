# Введение в безопасность AI

## 1. Актуальность
AI-системы становятся критическими компонентами инфраструктуры, но имеют уникальные уязвимости.

### Отличие от традиционной ИБ
Атаки направлены не на код или инфраструктуру, а на:
- Модель
- Данные  
- Процесс обучения

### Основные цели злоумышленников
1. **Нарушение конфиденциальности** — извлечение данных
2. **Нарушение целостности** — искажение работы
3. **Нарушение доступности** — отказ в обслуживании

---

## 2. Классификация угроз для AI (OWASP Top 10 для LLM)

| Категория | Описание |
|-----------|----------|
| Prompt Injection | Внедрение вредоносных инструкций в запрос для обхода ограничений модели |
| Insecure Output Handling | Небезопасная обработка вывода модели (выполнение кода, сгенерированного AI) |
| Training Data Poisoning | Отравление данных обучения для внедрения «бэкдоров» или искажения поведения |
| Model Denial of Service | Перегрузка модели через ресурсоемкие запросы |
| Supply Chain Vulnerabilities | Уязвимости в сторонних библиотеках, предобученных моделях и датасетах |
| Sensitive Information Disclosure | Раскрытие конфиденциальных данных через выходные данные модели |
| Insecure Plugin Design | Небезопасная архитектура плагинов и расширений |
| Excessive Agency | Чрезмерные полномочия AI-агента (доступ к системным функциям без ограничений) |

---

## 3. Жизненный цикл атак на ML-системы

1. **Сбор данных** – атаки на источники данных (подмена, перехват)
2. **Обучение** – отравление данных, внедрение триггеров
3. **Тестирование** – подбор метрик, маскирующих уязвимости
4. **Развертывание** – эксплуатация уязвимостей модели в продакшене
5. **Обслуживание** – атаки на обновления модели (poisoning через feedback)

---

## 4. Практические техники атак

### Adversarial Examples (состязательные примеры)
Добавление малозаметного шума к входным данным, изменяющего предсказание.

Примеры: атаки FGSM, PGD.

### Model Inversion
Восстановление элементов обучающей выборки по выходным вероятностям модели.

### Membership Inference
Определение, использовался ли конкретный образец в обучении (нарушение приватности).

### Backdoor Attacks
Внедрение скрытого паттерна, активирующего целевое поведение (например, всегда классифицировать как «одобрено»).

## 5. Методы защиты (Countermeasures)

### На этапе данных
- Аугментация с проверкой выбросов
- Дифференциальная приватность (DP-SGD)
- Криптографическое подписывание датасетов

### На этапе обучения
- Регуляризация, устойчивая к состязательным примерам (Adversarial Training)
- Использование ансамблей моделей

### На этапе инференса
- Валидация входных данных (фильтрация SQL-инъекций, PROMPT-инъекций)
- Мониторинг выходов модели (детекция аномалий)
- Rate limiting и аутентификация запросов

### Инфраструктурные
- Изоляция среды исполнения (sandboxing)
- Шифрование модели и данных at rest и in transit

---

## 6. Инструменты и фреймворки

- **CleverHans, Foolbox** – для генерации состязательных атак
- **TensorFlow Privacy** – для приватного обучения
- **ONNX, MLflow** – для управления моделями
- **OWASP Dependency-Check** – для проверки зависимостей

## 8. Рекомендации по безопасной разработке AI

1. **Shift-Left Security** – внедрять проверки безопасности на ранних стадиях (Data + Training)
2. **Минимизация привилегий** – модель не должна иметь доступ к базам данных или API без необходимости
3. **Постоянный мониторинг** – логирование всех запросов и ответов для обнаружения аномалий
4. **Обновление модели с учетом новых угроз** – регулярное переобучение на защищенных данных
5. **Прозрачность** – документирование всех решений по безопасности

---

## 9. Выводы

- Безопасность AI – это не одноразовое действие, а непрерывный процесс
- Большинство уязвимостей лежат не в математике модели, а в её интеграции и эксплуатации
- Практические навыки из курса помогают как атакующей, так и защитной стороне (red team / blue team)
